# Design Direction

本项目采用从 vibeui.top 选取的 **Mintlify** 方向作为设计参考：`documentation-as-product`，也就是把工具做成“可读、可信、可继续操作的产品化文档入口”，而不是后台管理系统或临时脚本页面。

参考文件保存在 `docs/design-references/vibeui-mintlify-DESIGN.md`，后续 UI 迭代必须先读本文件，再决定是否改动页面结构。

## 产品气质

- 成熟个人知识工作台：像一个认真维护的知识站点，而不是杂乱仪表盘。
- 中文阅读友好：标题短、摘要短、段落有层次，优先帮助用户快速判断“这是什么、为什么有用、下一步做什么”。
- 浅色内容区 + 深色首屏：首屏负责定位，内容区负责阅读与操作。
- Mintlify 式信息密度：卡片干净、留白充分、边界清楚、交互轻，不制造后台管理系统感。
- 默认不做后台管理系统：不堆表格、不堆按钮、不默认展示 JSON、不做伪控制台。

## 三个主操作

首页只保留三个真实操作：

- 添加资料：写入索引清单和 Obsidian 整理记录，不复制、不移动、不删除、不重命名源文件。
- 搜索回顾：做证据检索，返回匹配来源、摘要、推荐阅读顺序，不编造无来源答案。
- 生成 AI 上下文包：先预览候选来源，再确认生成 Markdown 和 prompt。

任何新功能如果不能归入这三条主线，应进入高级维护页，不能回到首页。

## 知识流

知识流不是文件列表，而是可持续阅读的知识卡片。每张卡片至少要回答：

- 这是什么：标题和类型。
- 有什么用：一句话结论。
- 从哪来：来源路径或来源说明。
- 下一步：可打开、可复制、可继续提取上下文。

卡片摘要不能直接暴露 Markdown 原文、frontmatter、路径噪音或执行日志。

## 知识详情

点击知识卡后展示 `知识详情`，结构固定为：

- 一句话结论：先给判断。
- 类型化正文：会话索引、知识卡片、普通笔记用不同段落结构。
- 来源：展示可追溯路径，并允许复制。
- 关联内容：展示与当前卡片共享主题的其他卡片。
- 可追问问题：引导用户继续复盘、沉淀、提取或交给 AI。

`关联内容` 和 `可追问问题` 是核心阅读体验，不能被隐藏到调试区。

## 视觉规则

- 主色使用深蓝黑作为首屏底色，内容区以白色和浅灰为主。
- 强调色优先使用 mint green，表达“已整理、可复用、可信”。
- 字号层级必须清楚：首屏大标题、section 标题、卡片标题、正文、路径说明。
- 卡片圆角和阴影要轻，避免玻璃拟态、Bootstrap 默认风、后台表格风。
- 结果区只展示用户能读懂的信息：做了什么、来源是什么、产物在哪、下一步能点什么。

## 禁止项

- 默认展示黑色代码框或大段 JSON。
- 把没有来源、没有产物的动作显示为“已完成”。
- 把浏览器拖放元数据说成真实本地目录扫描。
- 把 Codex 自己的内部指令、调试口吻、执行口吻展示给用户。
- 首页重新堆十几个按钮。

## 回归检查

每次 UI 或产品迭代后都要自问：

- 做到了视觉清晰阅读友好了吗？
- 做到了总结归纳到位，没有废话不遗漏关键信息了吗？
- 发给别人能看懂吗？传到 Codex 指引性足够吗？
- GUI 双向交互方便吗？信息内容容易编辑吗？
- 展示的方式足够方便吗？
- 信息集关联有做吗？能让人发散性思考吗？

检查结果要写入 `docs/iteration-logs/`，必要时同步写入 Obsidian 迭代记录。
