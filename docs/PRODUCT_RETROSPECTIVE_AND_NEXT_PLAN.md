# 产品复盘与下一步调整计划

## 当前结论

本项目已经从“文件扫描工具”和“按钮堆叠工作台”收敛为 **本地知识整理助手**。

它不应该替代 Codex，也不应该让用户误以为浏览器页面会自动接管所有电脑操作。GUI 的职责是把本地资料、Obsidian 笔记、AI 对话归档和历史报告整理成一个可看、可搜、可提取的知识入口；真正的执行能力由本地 Python action 层完成。

首页只保留三个真实主操作：

- **添加资料**：把文本、文件路径、目录路径或 AI 对话整理成新的 Obsidian 记录；路径只生成索引清单和 manifest，保留来源和判断理由。
- **搜索回顾**：从 Obsidian、助手报告、AI 对话归档和运行产物中做证据检索，返回摘要、来源、相关原因和下一步。
- **生成 AI 上下文包**：根据当前任务先预览候选来源，确认后再生成可复制给 AI 的上下文 prompt 和 Markdown 包。

`提醒` 保留为兼容命令和计划任务能力，但不再作为首页主功能。它只有在能稳定落盘、能解释来源、不会制造无意义“完成”结果时，才适合重新提升到主线。

## 过往要求沉淀

- 面向 Obsidian 新手：第一眼要知道这个工具是干什么的，不能被专业名词、指标和诊断入口淹没。
- 主线必须简单：整理、回顾、提取是当前核心；提醒作为次要自动化，不应该干扰日常入口。
- 本地优先：项目处理本机文件、Obsidian vault、AI 对话归档和本地报告；公开仓库只保存代码、文档、模板、测试和配置样例。
- 默认安全：不删除、不移动、不重命名、不重写源文件；写入只发生在新的 Obsidian 笔记或运行证据里。
- 操作必须真实：用户放入路径时要检查路径、扫描目标、记录来源；不能只把路径字符串写成“已完成”。
- 回顾不能编造：没有匹配来源就明确说明缺来源，并提示先添加资料或检查配置。
- 提取必须可追溯：上下文包必须包含来源路径、相关原因、压缩摘要、安全边界和建议给 AI 的下一步请求。
- GUI 不展示原始 JSON：默认结果只给人能读懂的“做了什么、来源是什么、产物在哪、下一步是什么”。
- 知识卡片只读：点击卡片只能展开详情和来源，不自动写入、不自动提取上下文。
- 生活 / 学习 / 工作分类要保留，但不能变成用户每天的额外负担。
- 历史资料不能丢：旧教程、旧报告、旧 action 和诊断能力先归档或降级到工具维护页，不在首页继续堆叠。
- 可迁移可复用：初始化、demo mode、配置样例、备份清单和验证脚本必须支持新机器恢复；个人知识库不进入公开仓库。
- UI 要像成熟个人知识站点：首屏简洁，内容区用知识卡片流沉淀，操作区少而清楚，不做后台管理系统风格。

## 产品边界

### GUI 是什么

GUI 是本地知识整理入口：

- 展示当前知识流。
- 提供三个主操作。
- 把输入、来源、产物和下一步显示清楚。
- 调用本地 action 层执行确定动作。

### GUI 不是什么

GUI 不是 Codex 本体，不是命令终端，也不是自动代理队列。

如果后续要让 Codex 或其他 AI 真正接管长任务，必须另做一个明确的执行队列，展示任务状态、日志、失败原因和人工确认边界，不能混在首页主操作里。

### Codex 在产品里的位置

Codex 适合做高阶执行者：读上下文包、理解真实文件、执行复杂修改、跑验证并汇报。

本工具适合做上下文资产层：把资料整理好、回顾出来、压缩成上下文包，让下一次 AI 对话不用重新从零读历史。

## 下一步调整计划

### 阶段 1：把三个主操作做实

目标：任何一次点击都不能是假完成。

- `添加资料` 要区分三种输入：文本、路径、AI 对话。
- 路径输入要先做本地目标检查，展示是否存在、是文件还是目录、可扫描数量和只读边界。
- 目录整理只生成清单、摘要和 Obsidian 新记录，不搬迁源文件。
- `搜索回顾` 要支持空结果、弱匹配和多来源结果三种状态。
- `生成 AI 上下文包` 要先把候选来源、相关原因和摘要拆开展示；确认后再保存 Markdown 产物和 prompt。

验收标准：

- 无来源时不能返回“完成”。
- 默认页面不出现原始 JSON。
- 每个结果都能看到来源路径或明确的缺来源原因。
- `organize / review / extract` 都符合统一 result schema。

### 阶段 2：把知识流做成可持续入口

目标：用户下滑看到的是自己已经积累的知识，而不是随机报告片段。

- 知识卡片按类型分组：入门、最近记录、项目、AI 对话、报告。
- 卡片标题要来自笔记标题或报告主题，描述要做短摘要，不能直接塞 Markdown 原文。
- 卡片详情面板显示摘要、来源、更新时间、标签、可执行下一步。
- 卡片点击保持只读；需要提取上下文时，用户再点明确的操作。

验收标准：

- 首页首屏只讲用途和三入口。
- 知识卡片不超过固定摘要长度。
- 卡片详情能复制路径或打开来源。
- 点击知识卡片不会产生写入。

### 阶段 3：把“给 AI 用”定义为导出能力

目标：AI 上下文包是清晰产物，不是含糊的“交接”。

- 输入当前任务。
- 从已整理内容中找来源。
- 生成上下文摘要。
- 生成给 AI 的下一步请求。
- 写入 Obsidian 和运行目录，方便复盘。

验收标准：

- 输出同时包含 Markdown 包和可复制 prompt。
- 每条上下文都有来源路径。
- prompt 里必须包含任务目标、相关来源、执行边界和验收标准。
- 无匹配来源时提示先添加资料或扩大检索范围。

### 阶段 4：把迁移和备份做成基础设施

目标：新机器 clone 后能跑通 demo，再接入真实 vault。

- 初始化脚本检查 Python、配置、demo vault、GUI 启动和验证命令。
- `config.example.json` 只放可公开样例。
- `config.local.json` 只留本机路径并保持忽略。
- 备份清单记录 commit、恢复命令、忽略文件说明和个人 vault 备份边界。

验收标准：

- demo mode 能完整跑通三个主操作。
- 公开仓库不包含个人路径、真实 vault 内容和私有运行报告。
- `verify-harness` 能检查本地与远端状态。

## 当前不做

- 不做源文件自动搬迁。
- 不做语义向量库。
- 不做外部通知作为核心能力。
- 不把个人 Obsidian vault 放进公开仓库。
- 不在首页展示低频诊断工具。
- 不把浏览器 GUI 包装成 AI 自动执行器。

## 质量门禁

后续任何改动都必须过以下检查：

- README 和 GUI 继续只主推 `添加资料 / 搜索回顾 / 生成 AI 上下文包`。
- 首页不能回退成大量按钮和诊断入口。
- 默认结果不能展示原始 JSON。
- 无来源、无路径、无产物时不能显示“已完成”。
- 所有写入都只能写新笔记或新运行证据。
- 知识卡片点击保持只读。
- GUI E2E 必须覆盖三入口、知识卡片详情和高级页边界。

## 最近一轮推荐执行顺序

1. 先补 `添加资料` 的输入判定和路径检查结果展示。
2. 再修 `搜索回顾` 的空结果和弱匹配状态，不让它假完成。
3. 再增强 `生成 AI 上下文包` 的来源选择和 Markdown 产物展示。
4. 最后整理知识流摘要质量和卡片详情交互。
5. 每一步都跑单元测试、GUI E2E 和质量门禁，再决定是否继续压缩高级页。
