# 2026-05-11 vibeui Mintlify 产品优化迭代自查

## 触发背景

用户要求读取新的 Codex 全局准则，并从 vibeui.top 选择一个合适的 `design.md` 跑一遍产品优化。

新的全局准则强调：需求、设计和计划阶段要主动追问与反驳；执行阶段必须读取真实内容、少废话、能做就做。本轮目标明确，所以直接执行，不停留在方案讨论。

## 设计参考

- 来源：`https://vibeui.top/design-md/mintlify/DESIGN.md`
- 本地归档：`docs/design-references/vibeui-mintlify-DESIGN.md`
- 选择理由：Mintlify 更接近 `documentation-as-product`，适合把本项目从“工具按钮集合”收敛成“可读、可追溯、可继续操作的知识整理站点”。

## 本轮改动

- 新增 `DESIGN.md`，固化 Mintlify 风格如何落到本项目：三主操作、知识流、知识详情、关联内容、可追问问题、禁用伪控制台。
- 扩展 `knowledge_feed` 数据结构：知识卡现在包含 `detail_kind`、`detail_sections`、`thinking_prompts`、`related_items`。
- 优化 GUI 知识详情：不再只展示粗糙摘要，改为一句话结论、类型化段落、来源、关联内容、可追问问题。
- 增加质量门：`project_quality.py` 检查 `DESIGN.md` 和 vibeui 参考文件是否存在，并确认核心设计词没有丢失。
- 补测试先行：为知识详情结构、HTML 展示函数和设计参考质量门写回归测试。

## 九问自查

- 做到了视觉清晰阅读友好了吗？基本做到。详情页从散乱摘要改成卡片式阅读结构，强调色也从泛蓝改成 Mintlify 式 mint green。
- 做到了总结归纳到位，没有废话不遗漏关键信息了吗？部分做到。会话索引和知识卡片已有类型化摘要，但普通笔记仍依赖 Markdown 标题结构，后续要继续优化提炼质量。
- 发给别人能看懂吗？基本能。新增 `DESIGN.md` 解释产品气质、主操作、知识详情和禁用项。
- 传到 Codex 指引性足够吗？比之前更足。知识详情增加可追问问题，能引导下一步让 Codex 读取来源、沉淀卡片或生成上下文包。
- GUI 双向交互方便吗？本轮主要改善阅读和展示；输入闭环仍依赖三主操作后端，后续要继续做真实路径扫描、候选预览和错误状态。
- 信息内容容易编辑吗？部分做到。来源路径明确，可回到 Obsidian 原文编辑；GUI 内联编辑未做。
- 展示的方式足够方便吗？比之前更方便。点击卡片后能看到来源、关联、追问，不需要读 Markdown 原文。
- 信息集关联有做吗？已做第一版。通过标题、摘要、标签和路径关键词给出相关卡片，但不是语义检索。
- 能让人发散性思考吗？已做第一版。`可追问问题` 会引导用户继续复盘、沉淀或提取上下文。

## 下一轮改进

- 把 `关联内容` 从关键词重叠升级为更稳定的主题规则，但暂不引入向量库。
- 给每类知识卡设计更严格的摘要模板，减少普通笔记摘要退化成原文摘录。
- 用 Playwright 对知识卡展开、关联内容、可追问问题做截图回归。
