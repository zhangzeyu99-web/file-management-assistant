# GUI 交互说明

GUI 是本地知识整理助手的站点式知识库入口，不是临时工具页，也不是命令输出页。它负责展示知识流、跳转三个主操作，并把输入、来源、产物和下一步展示清楚；实际动作由 `knowledge_assistant.py` 的 `organize`、`review`、`extract` 执行。

## 页面结构

- 首屏：深色 hero，只展示产品名、一句话定位、安全边界和两个 CTA。
- 三张主卡：`添加资料`、`搜索回顾`、`生成 AI 上下文包`，点击后跳转到同页锚点区。
- 知识流：基于已有 Obsidian 笔记、AI 归档、历史报告和运行产物生成卡片；点击卡片只展开详情和来源。
- 三个操作区：每个区只保留一个轻量输入、一个主按钮和一个结果卡。
- 结果卡：默认展示人能读懂的结果，并固定拆成 `做了什么`、`来源是什么`、`产物在哪`、`下一步`。
- 工具维护页：低频诊断入口从首页移到 `/advanced`，包括 `file-radar`、`obsidian-health`、`legacy-index` 和本地资料打开入口。

## 产品化交互原则

- 首屏极简：首页只回答“这是什么、有什么用、从哪里开始”，不堆全部按钮。
- 流式浏览：知识内容以标题 + 描述卡片持续沉淀，先浏览，再决定是否操作。
- 搜索/筛选：回顾和提取先通过关键词/任务缩小范围，再显示来源和产物，避免把全部候选当成今日任务。
- 来源可追溯：结果区优先显示匹配来源、最近来源和产物路径，便于继续交给 AI 使用。
- 安全默认：搜索回顾和诊断默认只读；添加资料和生成 AI 上下文包只写新记录。

## 主流程

```text
进入首页 -> 点击三张主卡跳到锚点区 -> 输入资料或问题 -> 查看结果卡 -> 需要诊断时进入工具维护页
```

## 可点击入口

| 按钮 | 输入 | 实际 action | 结果 | 文件影响 |
| --- | --- | --- | --- | --- |
| 知识卡片 | 无 | 无 | 展开标题、描述、来源路径、建议动作 | 只读 |
| 添加资料 | 文本 / 路径 / AI 对话 | `organize` | Obsidian 新整理记录、路径索引清单、来源、领域判断、下一步 | 只写新笔记和清单 |
| 搜索回顾 | 关键词或问题 | `review` | 本地摘要、匹配来源、相关原因、可打开路径 | 只读 |
| 生成 AI 上下文包 | 当前任务 | `extract` | 先预览候选来源，确认后生成 prompt 和 Markdown | 预览只读，确认后写新上下文包 |
| 检查本地目标 | 本地路径 | `inspect-local-targets` | 路径是否存在、数量、模式 | 只读 |
| 打开工具维护页 | 无 | 页面跳转 `/advanced` | 进入诊断与资料入口 | 只读 |
| 查看文件雷达 | 可空或路径 | `file-radar` | 本地文件报告 | 工具页执行，写新报告 |
| 检查知识库 | 可空 | `obsidian-health` | vault 体检报告 | 工具页执行，写新报告 |
| 二次整理旧资料 | 可空 | `legacy-index` | 旧资料索引 | 工具页执行，写新索引，不移动旧文件 |
| 打开 Obsidian / 教程 / 交互说明 | 无 | `open-obsidian` / `open-guidebook` / `open-interaction-guide` | 打开本地资源 | 不改知识库 |

输入不足时不会硬写占位记录：没有资料就不生成整理笔记；路径不存在或只有浏览器文件元数据时不生成假整理记录；没有匹配来源就不生成假上下文包；没有结论或可读取来源就不生成知识卡。

## 本地文件 / 目录目标

真实扫描本机文件时，优先粘贴完整路径，例如：

```text
D:\Downloads
D:\Work\report.md
```

拖放和选择文件主要用于告诉 GUI “这些是上下文来源”。浏览器不能直接把本机完整目录权限交给后端，所以需要扫描时仍建议粘贴路径。

## 安全边界

默认不删除、不移动、不重命名、不重写源文件。添加资料只做索引清单和新记录；生成 AI 上下文包必须先预览来源，确认后才写新的 Obsidian 笔记和运行证据；工具维护页的诊断动作只生成报告或打开本地资源。
