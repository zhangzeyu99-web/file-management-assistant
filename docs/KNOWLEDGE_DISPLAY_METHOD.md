# 知识展示整理方法论

## 一句话结论

知识展示不再直接露出原始 Markdown、frontmatter 或长路径墙；统一改成“结论先行、来源可追溯、渐进展开、下一步可执行”的阅读卡片。

## 为什么这样整理

- 用户进入知识流时通常只想快速判断“这张卡有没有用”，所以列表卡只显示标题、类型、时间和一句话结论。
- 用户点击查看时才需要更多上下文，所以详情页按文档块展开：适用场景、关键结论、下一步、来源。
- AI 续用需要真实来源和边界，所以来源路径必须保留，但不能压过正文阅读。
- Obsidian 新手不应被复杂双链、标签系统和原始元数据淹没，先用稳定模板保证可读、可搜、可复用。

## 展示层结构

| 层级 | 显示内容 | 目的 |
| --- | --- | --- |
| 知识流卡片 | 标题、类型、更新时间、一句话结论 | 快速扫读，判断是否值得打开 |
| 阅读详情 | 一句话结论、适用场景、关键结论、下一步、来源 | 像飞书文档块一样阅读，减少噪音 |
| 来源区 | 来源名称、真实路径、复制按钮 | 可追溯、可回到原文、可给 AI 使用 |
| 高级调试 | 原始 JSON 和诊断信息 | 只给维护者，不默认展示给用户 |

## Obsidian 落档模板

```markdown
# 标题

类型：Card
领域：工作 / 学习 / 生活
来源：来源路径或来源说明
创建时间：`YYYY-MM-DD HH:mm:ss +0800`

> 一句话结论：先用一句话说明这张卡以后怎么帮到我。

## 适用场景

- 什么时候应该打开这张卡。
- 什么任务可以复用它。

## 关键结论

- 可复用规则 1。
- 可复用规则 2。
- 必要边界或例外。

## 下次怎么用

- 搜索哪些关键词能找到它。
- 生成 AI 上下文包时是否应作为来源。

## 来源

- 来源名：`D:\path\to\source.md`

## 验收标准

- 能被下一次任务直接引用。
- 保留来源，不覆盖原始资料。
```

## 写作规则

- 标题短，先放用户会搜索的关键词。
- 每段只解决一个问题，避免连续大段文字。
- 关键结论用列表，不写流水账。
- 来源路径只放在来源区，正文里不堆路径。
- 如果没有证据来源，不显示“已完成”，只显示缺来源或待补充。

## 参考方法

- Microsoft Style Guide：短标题、短段落、重要信息前置、建立一致内容模式。
- VA.gov Design System：用层级标题、列表和可折叠/分组内容降低阅读负担。
- Cornell Note Taking：保留“线索、笔记、摘要”的思想，但落到本项目里改成“场景、结论、来源、下一步”。
- 飞书文档格式经验：用标题目录、列表、引用块和表格形成块状阅读，不用花哨样式制造噪音。
